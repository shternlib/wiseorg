# Intenture - Known Vulnerabilities & Future Roadmap

## Статус
- **Document Type:** Vulnerability register & roadmap
- **Target versions:** v1.4+
- **Приоритет:** решать после стабилизации v1.3

---

## Vulnerability 1: Масштабируемость Canvas для крупных замыслов

### Описание
Canvas - плоская структура. Для простых замыслов это работает. Но для крупных проектов (WiseTeam как продукт) замысел содержит десятки подзамыслов: архитектура, UI, агенты, IDL, бизнес-модель, маркетинг. Плоский Canvas не может удержать эту сложность.

### Почему это уязвимость
При реализации "WiseTeam разрабатывает WiseTeam" PM-агент получит intenture такого объема, что плоская структура Canvas станет нечитаемой и для человека, и для AI. Потеряется целостность замысла при декомпозиции.

### Направление решения
Иерархический Canvas: root intenture содержит ссылки на под-intenture (sub-intenture), каждый из которых имеет собственный Canvas, но наследует Constraints и Value от родителя. Нужна модель:
- **intenture contains sub-intenture** (новая связь в Meta-Model)
- **Constraint inheritance** - как Constraints родителя применяются к потомкам
- **Value aggregation** - как Value под-замыслов складывается в Value родителя
- **Readiness roll-up** - родительский intenture не может быть Realizable, пока blocking sub-intenture не в Realizable

### Влияние на язык
Потребуется расширение Meta-Model, Canvas, AI-facing form, Visual Notation (диаграмма декомпозиции).

### Приоритет
Критический для WiseTeam. Нужен до начала полноценной самореализации платформы.

## Vulnerability 2: Нет формата для мульти-intenture проектов

### Описание
Если у человека пять связанных замыслов (платформа, маркетинг, монетизация, партнерства, найм), каждый - отдельный intenture, но они зависят друг от друга. Язык не описывает связи между intenture.

### Почему это уязвимость
Без связей между intenture PM-агент будет оптимизировать каждый замысел изолированно. Изменение в одном intenture (например, смена бизнес-модели) должно каскадно повлиять на другие (маркетинг, найм). Без формальных связей эти зависимости теряются.

### Направление решения
Intenture Graph - граф связей между intenture:
- **depends_on** - этот intenture не может быть реализован без другого
- **enables** - реализация этого intenture открывает возможность для другого
- **conflicts_with** - эти intenture содержат потенциальный конфликт
- **shares_constraint_with** - общие ограничения
- **shares_actor_with** - общие участники

Нужен отдельный артефакт: **Intenture Portfolio Map** - визуализация графа связей.

### Влияние на язык
Новый раздел в Meta-Model (inter-intenture relations), новый тип диаграммы в Visual Notation, расширение Linked Artifacts.

### Приоритет
Высокий. Становится актуальным, как только человек работает более чем с одним intenture одновременно.

## Vulnerability 3: Assumed by AI как скрытый risk

### Описание
Если AI пометил блок Canvas "Assumed by AI", но человек не заметил этот статус, реализация может пойти по ложному пути. Нет формального правила, запрещающего переход в Realizable с неподтвержденными допущениями.

### Почему это уязвимость
В реальной работе человек может не перечитать весь Canvas перед подтверждением старта. Одно неверное допущение AI (например, о юрисдикции или о допустимых действиях агента) может привести к серьезным последствиям.

### Направление решения
Жесткое правило: **ни один блок со статусом "Assumed by AI" не может существовать при переходе intenture в Realizable**. Все допущения должны быть:
- либо подтверждены человеком (статус меняется на Answered)
- либо явно приняты человеком как допущения с маркировкой "Accepted Assumption" (новый статус)
- либо переведены в Unknown Yet (блокирующий)

PM обязан показать человеку список всех Assumed by AI блоков перед Readiness Decision и получить явное подтверждение по каждому.

### Влияние на язык
Обновление статусов Canvas (новый статус Accepted Assumption), обновление Readiness Checklist, обновление AI Interpretation Protocol (шаг 7).

### Приоритет
Высокий. Это вопрос safety.

## Vulnerability 4: Нет механизма отката (rollback)

### Описание
Explication Record Versioning Policy описывает создание новых версий, но не описывает процедуру отката к предыдущей версии. Если реализация по текущему intenture пошла не туда, нет формального способа вернуться к предыдущему snapshot и начать заново.

### Почему это уязвимость
В Evolution Mode замысел может измениться так, что текущая версия окажется хуже предыдущей. Без формального rollback PM-агент не знает, можно ли откатить и как именно.

### Направление решения
Добавить в Versioning Policy:
- **Rollback trigger** - когда допустим откат (Critical Signal, провал метрик, решение человека)
- **Rollback procedure** - PM восстанавливает предыдущий Explication Record как текущий, помечает отмененную версию как "Reverted"
- **Rollback constraints** - если артефакты реализации уже созданы по текущей версии, нужен план отката артефактов

### Влияние на язык
Расширение Versioning Policy, новый статус Explication Record ("Reverted"), обновление Lifecycle Model (переход через rollback).

### Приоритет
Средний. Становится важным при длительных проектах с множеством итераций.

## Vulnerability 5: Нет формата для временных и ресурсных ограничений

### Описание
Constraint Type Vocabulary (запланирован в v1.3) включает budget и timeline, но нет формального способа записать: "бюджет - $5000", "дедлайн - 1 мая 2026". Constraints сейчас - текст. Для AI, управляющего реализацией, числовые ограничения критичны.

### Почему это уязвимость
PM-агент, координирующий работу AI-команды, должен трекать расход бюджета и соблюдение сроков. Если бюджет и timeline не формализованы, контроль невозможен.

### Направление решения
Расширение Constraint с формализованными подтипами:
- **Budget Constraint** - currency, amount, period, flexibility
- **Timeline Constraint** - deadline, milestones, flexibility
- **Resource Constraint** - type, quantity, availability

Формат:
```
Constraints:
  - [budget] $5000 total, flexible +20%
  - [timeline] MVP by 2026-05-01, hard deadline
  - [safety] программа не должна вредить человеку
```

### Влияние на язык
Расширение Constraint в Meta-Model, обновление AI-facing form syntax.

### Приоритет
Высокий для Product/Service и AI Agent паттернов. Менее критичен для Health Care.

## Vulnerability 6: Нет формата для приоритизации Value

### Описание
Intenture может создавать несколько Value для разных Actors. Но нет способа указать приоритет: какая ценность важнее. Если ресурсы ограничены (а они всегда ограничены), AI должен знать, что оптимизировать в первую очередь.

### Почему это уязвимость
При реализации AI-команда будет принимать trade-off решения. Без явной приоритизации Value каждый агент будет оптимизировать свою часть, что может разрушить баланс.

### Направление решения
Ввести **Value Priority** - упорядоченный список Values от наиболее важной к наименее. Или числовой вес. Или группировку: must-have / should-have / nice-to-have.

### Влияние на язык
Расширение Value в Meta-Model, обновление Canvas (Value block получает priority), обновление AI-facing form.

### Приоритет
Средний. Становится критичным при сложных мульти-actor замыслах.

## Vulnerability 7: Отсутствие формальной валидации AI-facing form

### Описание
Сейчас AI-facing form - текст. Нет schema, нет формальной валидации. PM-агент не может автоматически проверить: все ли обязательные поля заполнены, нет ли пустых Critical Blocks, корректны ли статусы.

### Почему это уязвимость
При масштабировании (множество intenture, множество версий) ручная проверка формы невозможна. Нужна машиночитаемая schema.

### Направление решения
Создать **Intenture Schema** - JSON Schema или аналогичный формат для валидации AI-facing form:
- обязательные поля
- допустимые значения статусов
- правила полноты (все Critical Blocks должны быть Answered для Realizable)
- правила непротиворечивости

### Влияние на язык
Новый companion artifact: Intenture JSON Schema. Возможно, альтернативный формат AI-facing form в JSON/YAML вместо свободного текста.

### Приоритет
Средний для пилота. Критичный для production-уровня платформы.

## Рекомендуемый порядок решения

**v1.4 (ближайшая после 1.3):**
- V3 (Assumed by AI) - вопрос safety, решается быстро
- V1 (Иерархия Canvas) - критичен для WiseTeam
- V2 (Мульти-intenture) - естественное продолжение V1
- V5 (Ресурсные ограничения) - нужен для реального управления проектом

**v1.5:**
- V4 (Rollback) - нужен после накопления опыта эволюции замыслов
- V6 (Приоритизация Value) - нужен для сложных мульти-actor замыслов
- V7 (JSON Schema) - нужен для production-уровня

---

## Связь с WiseTeam

Уязвимости V1, V2 и V5 напрямую блокируют полноценную самореализацию WiseTeam. Замысел "WiseTeam разрабатывает WiseTeam" - это крупный иерархический мульти-intenture проект с конкретным бюджетом и timeline. Без решения этих уязвимостей PM-агент не сможет полноценно управлять таким проектом через Intenture.
